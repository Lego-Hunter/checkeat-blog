# 데이터 인벤토리 먼저, 화면 설계는 그 다음

> **상태**: 초안 시드 (2026-03-10 세션 기록 기반, 집필 필요)

## 이 글을 읽으면 알 수 있는 것

- DB에 이미 있는 데이터를 전수 파악한 뒤 UI를 설계하는 "데이터 퍼스트 UX" 접근법
- 100만 건 테이블에서 필드별 커버리지를 빠르게 측정하는 방법
- 커버리지 수치가 UI 의사결정을 바꾼 실제 사례

---

## 핵심 소재

### 배경: "프론트엔드 MVP를 시작하자"에서 멈춘 이유

프론트엔드 MVP 착수 직전, 사용자(프로젝트 오너)가 핵심 질문을 던졌습니다.

> "사용자에게 제공할 수 있는 우리의 정보가 어떤 것들이 있는지 세밀하게 파악한 뒤에, 섬세하게 배치를 기획하는 게 UX 차원에서도 디자인 차원에서도 중요하다는 생각이 들어서요."

AI(구현자)는 이미 구현된 코드를 기반으로 "어디를 고칠까"를 생각하고 있었습니다. 사용자는 한 단계 앞서, "뭘 보여줄 수 있는지부터 파악하자"고 말한 것입니다.

### 발견: 14,744건의 제품 이미지가 100% 있었지만 아무 데도 안 보이고 있었다

데이터 인벤토리 실행 결과, 고가치 미노출 데이터가 대량 발견:

| 데이터 | 건수 | 상태 |
|--------|------|------|
| HACCP 제품 이미지 (imgurl1/2) | 14,744건 100% | serializer에 있으나 **프론트 미사용** |
| 고열량저영양 구분 | 101,823건 (9.8%) | serializer에 있으나 **프론트 미사용** |
| HACCP 알레르기 텍스트 | 6,221건 (42.2%) | 이미 구현됨 (양호) |
| raw_material_summary (GMO/원산지) | serializer 구현 완료 | **프론트 타입조차 미정의** |
| 첨가물 사용기준 | 743개 | API만 존재, 프론트 없음 |

(실제로는 이미지는 이전 세션에서 이미 구현되어 있었음. 그러나 "이미 구현된 것도 전수 파악 대상에 포함"하는 것이 인벤토리의 핵심.)

### 방법: SQL COUNT + NULLIF 패턴

100만 건 테이블에서 필드별 non-null/non-empty 비율을 한 쿼리로:

```sql
SELECT
  COUNT(*) as total,
  COUNT(NULLIF(product_name, '')) as product_name,
  COUNT(NULLIF(hieng_lntrt_dvs_nm, '')) as hieng_lntrt_dvs_nm,
  -- ...
FROM product
```

주의점: boolean 필드는 `= true` 비교가 텍스트 컬럼에서 에러 발생 (실제로 겪음).

### 의사결정: 커버리지 수치가 UI를 바꿨다

- **9.8% (고열량저영양)**: "값이 있는 제품에서만 배지 표시 + 툴팁으로 설명" (Option C)
  - 90.2%에 빈 배지를 붙이면 노이즈 → 기각
  - 검색 결과 상단에 매번 안내 문구 → 반복적 → 기각
- **0% (purpose_text, production_end_yn)**: 필드 자체가 비어있음 → UI에서 완전 제외
- **61.3% (HACCP nutrient)**: 있으면 우선 표시, 없으면 FoodNutrition 폴백
- **0건 (GMO=Y)**: GMO 배지 기능은 구현하되, 현재 데이터에서는 표시될 일 없음 → 향후 데이터 입수 시 자동 활성화

### 교훈

1. **"이미 구현된" 코드도 데이터 관점에서 재점검해야 한다** — serializer에 필드가 있어도 프론트에서 안 쓰면 사용자에게는 없는 것과 같다.
2. **커버리지 0%인 필드를 UI에 넣으면 빈 공간만 생긴다** — 데이터가 실제로 있는지 먼저 확인.
3. **커버리지가 낮아도 "있는 곳에서만 표시"하면 가치가 있다** — 9.8%인 고열량저영양도 해당 제품에서는 핵심 정보.
