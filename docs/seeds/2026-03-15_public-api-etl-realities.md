## 소재 요약

ETL 코드 리뷰 18건 전수 처리 과정에서 발견한 두 가지 공공 API 현실:
1. **워터마크(증분 적재)가 불가능한 이유** — 식약처 API가 날짜 필터를 지원하지 않고, CHNG_DT/LAST_UPDT_DTM 필드의 신뢰성도 보장되지 않아 데이터 누락 리스크가 성능 이득을 초과
2. **괄호 파서를 "관대하게" 만들어야 하는 이유** — 97.8만 건 전수 조사 결과 IUPAC 화학명에서 `[`로 열고 `)`로 닫는 표기 오류 7건 발견. 엄격 적용 시 후속 원재료가 연쇄 합쳐지는 치명적 부작용

### 핵심 발견
- 공공 API는 증분 적재를 위한 인프라(날짜 필터, 변경 로그)를 제공하지 않음
- 대안으로 해시 기반 skip이 더 안전하지만, 현재 규모(~115만 건)에서는 PostgreSQL upsert 자체가 충분히 효율적
- 파서 설계 시 "데이터가 완벽하다"는 가정은 위험 — 관대한 처리(lenient parsing)가 정확도를 높이는 역설

## story 관점 (운영/브랜드)

**제목안**: "104만 건 식품 데이터, 매일 전부 다시 읽는 이유"

- 도입: "증분 적재하면 빠르지 않나요?" — 당연한 질문
- 전개: 식약처가 과거 데이터를 수정일 변경 없이 업데이트할 수 있음. 한 건이라도 누락되면 사용자가 잘못된 성분 정보를 봄
- 결론: 속도보다 정확성. "매일 전부 읽는다"는 비효율이 아니라 신뢰의 비용
- 메시지: 성분 해석 서비스에서 "빠르지만 가끔 틀리는 것"은 선택지가 아님

## tech 관점 (기술/프로세스)

**제목안**: "공공 API ETL 삽질기: 워터마크와 관대한 파서"

- **워터마크 불가 분석**: API 날짜 필터 미지원 + 필드 신뢰성 미보장 + 페이지네이션 중 데이터 변경 리스크
- **대안 비교**: 워터마크 skip vs 해시 기반 skip vs 전체 적재 (트레이드오프 매트릭스)
- **관대한 파서의 역설**: 97.8만 건 중 7건(0.0007%)의 괄호 불일치가 엄격 파서를 망가뜨리는 실증
  - 현행(관대): `[`로 열고 `)`로 닫아도 depth 감소 → 12토큰 정상 분리
  - 엄격: depth 미해소 → 수크랄로스, 프로필렌글리콜까지 합쳐져 10토큰 (2개 성분 손실)
- **교훈**: 공공 데이터는 "99.9993% 정상"이지만, 그 0.0007%가 파서 설계를 결정함
