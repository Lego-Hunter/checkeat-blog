# 글감 시드: 데이터 인벤토리 먼저, 화면 설계는 그 다음

> 날짜: 2026-03-10
> 상태: 시드
> 소스: checkeat_project 데이터 인벤토리 기반 프론트엔드 UI 배치 (커밋 28012fd)

## 소재 요약

프론트엔드 MVP 착수 직전, "뭘 보여줄 수 있는지부터 파악하자"는 사용자의 제안으로 100만 건 테이블의 필드별 커버리지를 전수 측정. 그 수치가 UI 의사결정을 직접 바꿨다 — 9.8%인 고열량저영양은 "있는 곳에서만 배지 표시", 0%인 필드는 UI에서 완전 제외, 14,744건의 제품 이미지는 serializer에 있었지만 프론트에서 안 쓰고 있었음을 발견.

## OSMU 활용

### story (운영 블로그) — 후보

**관점**: AI 두 마리와 UX를 기획하다 — 비개발자가 "데이터부터 보자"고 방향을 틀어준 순간.

- AI(구현자)는 "어디를 고칠까"를 생각하고 있었는데, 사용자가 한 단계 앞서 "뭘 보여줄 수 있는지"를 물었다
- Claude가 브리핑 작성, Gemini가 감리, 사용자가 최종 판단하는 3자 협업
- 커버리지 9.8%인 데이터를 버릴 것인가, 살릴 것인가 — Option A/B/C 논쟁
- "데이터가 있다"와 "사용자가 볼 수 있다"는 전혀 다른 문제

**연결 회차**: 12화(새우깡), 17화(AI 두 명이 그만하세요), 21화(여기까지)

### tech/ai-dev — 08화 후보

**관점**: DB에 이미 있는 데이터를 전수 파악한 뒤 UI를 설계하는 "데이터 퍼스트 UX" 접근법.

- 100만 건 테이블에서 SQL COUNT + NULLIF 패턴으로 필드별 커버리지 측정
- boolean 필드 `= true` 비교가 텍스트 컬럼에서 에러 발생 (실제 삽질)
- 커버리지 수치별 UI 배치 규칙: 0% → 제외, 9.8% → 조건부 표시, 61.3% → 우선+폴백, 100% → 항상 표시
- 14,744건 이미지가 serializer에는 있었지만 프론트 미사용 — "구현된 것도 인벤토리 대상"

**연결 글**: 01화(새우깡 사건), 06화(99.5%에서 멈추다), 07화(검증 체계)
