# 글감 시드: 하루에 네 번 점프 — 아카이브 이식, 프론트엔드 UX, 서버 이사, 데이터 이전

> 날짜: 2026-03-10
> 상태: 시드
> 소스: checkeat_project + checkeat-infra 2026-03-10 전체 작업 (총괄 관점)

## 소재 요약

하루 동안 성격이 전혀 다른 세 가지 작업을 연속으로 수행:

1. **아카이브 이식**: 6개월 전 기획 문서 230개 태스크에서 15개를 추려 현행 코드와 대조, 8개 이미 구현 확인, confidence 임계값 폐기 결정
2. **프론트엔드 UX 3종 구현**: Popover 컴포넌트, 온보딩 4단계 확장, ConfidenceBadge→MatchMethodBadge 전환
3. **AWS 마이그레이션**: Railway→EC2+RDS, Terraform IaC, PG17→PG16 DB 이전, SSL 설정, 비용 $0
4. **데이터 이전 + 자동화**: RDS 전체 적재(5.2GB 무손실), ETL cron EC2 이전, Vercel Git 자동배포 연결

네 작업의 공통점: 모두 "지금 하지 않으면 안 되는" 타이밍 압박이 있었다 — 아카이브는 잊혀지기 전에, 프론트엔드는 MVP 데드라인, 서버는 Railway 만료 D-7, 데이터는 서버 이사 직후.

## OSMU 활용

### story (운영 블로그) — 22~23화 연결 소재

**관점**: 비개발자가 하루에 기획 정리, 화면 구현, 서버 이사를 모두 해낸 이야기. 세 가지가 어떻게 연결되는지.

- 아침: 6개월 전의 나와 대화 — 230개 할 일 목록 중 자연스럽게 도달한 것들
- 낮: confidence라는 숫자가 실데이터 앞에서 무너지고, match_method로 대체한 과정
- 저녁: "서버를 옮긴다"는 말이 구체적으로 뭘 뜻하는지 처음 체감
- 밤: "덤프가 뭐에요?"에서 5.2GB 무손실 이전 + ETL 자동화까지
- 관통 주제: **타이밍** — 완벽할 때까지 기다리면 기회가 사라진다

**연결 회차**: 21화(여기까지) 이후 첫 "다시 시작" 에피소드

### tech — 크로스 시리즈 소재

**관점**: 하루 안에 서로 다른 기술 영역(데이터 분석 → 프론트엔드 → 인프라)을 오간 의사결정 흐름.

- 아카이브 이식의 판정 프레임워크: "이미 구현 / 재설계 후 구현 / 장기 보류 / 폐기" 4단계 분류
- confidence 이산값 발견 → MatchMethodBadge 설계까지의 기술적 경로
- Popover 컴포넌트: native title → click/tap 전환의 UX 근거 (모바일 hover 불가)
- Terraform SSH 키 자동 생성, PG 버전 마이그레이션 삽질
- AI 역할 변화: 같은 날 안에서 "기획 감사자" → "프론트엔드 구현자" → "인프라 페어"

**연결 글**: 개별 시드 3개(archive-transplant, data-first-ux, railway-to-aws-migration)의 상위 맥락
