# AI 감리자의 사각지대 — 코드를 못 읽는 감리자와 일하기

- **상태**: seed
- **발견일**: 2026-03-11
- **관련 커밋**: Sprint J-3 (프로필 UX 리팩토링 및 점진적 발견)

## 소재 요약

Sprint J-3에서 "점진적 발견(Progressive Disclosure)" 기능을 구현할 때, Gemini(감리자)가 `doesProductContainRule`이라는 **존재하지 않는 함수**를 기반으로 구현 제안을 보냈다. Claude(구현자)가 실제 코드베이스를 확인하고 "그 함수는 없으며, 기존 `matchRiskTags`를 재사용하는 것이 낫다"고 반론하여 채택됨.

이후 사용자가 Gemini에게 "코드를 안 읽어 보고 답변하는 건 아니냐"고 직접 지적. Gemini는 "컨텍스트에 포함된 파일만 인지하며, 전체 코드베이스를 읽지 않는다"고 솔직하게 인정. 이를 "버그가 아니라 의도된 워크플로우"라고 설명함.

## story (운영 블로그) 관점

### 제목 후보
- "AI 감리자에게 코드를 안 읽어봤냐고 물었더니"
- "두 AI의 역할 분담에서 발견한 협업의 본질"

### 글 구조 스케치
1. 상황: 두 AI(Claude=구현, Gemini=감리) 체제로 프로젝트 운영 중
2. 사건: 감리자가 없는 함수를 제안 → 구현자가 반론
3. 발견: 사용자가 "코드를 안 읽어봤냐" 직접 질문 → 솔직한 답변
4. 교훈: AI 감리자의 가치는 "전략적 방향 제시"이지 "코드 세부 암기"가 아님
5. 실전 팁: 감리자 제안은 반드시 실제 코드와 대조 검증 필요

## tech (기술 블로그) 관점

### 제목 후보
- "AI 에이전트 협업에서 컨텍스트 비대칭 문제"
- "LLM 감리자의 hallucination을 구조적으로 방지하는 법"

### 글 구조 스케치
1. 문제 정의: LLM은 컨텍스트 윈도우에 주어진 정보만 인지 → 코드베이스 전체 인식 불가
2. 실제 사례: `doesProductContainRule` hallucination
3. 구조적 해법: 역할 분리 (전략 vs 구현) + 교차 검증 프로세스
4. auditor_guidelines.md에 반영된 규칙
5. Claude Code vs Gemini의 도구 접근 차이 (파일 R/W vs 붙여넣기)
